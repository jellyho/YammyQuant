YammyQuant

An agentic quant research cockpit. Claude Code is the operator — it drives a CLI toolbelt over the yammyquant library to collect data, backtest, train RL agents, generate signals, and run a paper/live portfolio. A web dashboard lets you watch everything live and leave instructions.

no paid LLM API — the brain is your Claude Code session paper-by-default, two-gate live trading KR + crypto + ccxt exchanges

There is no paid inference in the loop. The dashboard is the cockpit; the conversation happens in Claude Code. Everything the operator does is written to shared state, so the dashboard reflects it in real time — and survives across sessions.

Source: github.com/jellyho/YammyQuant

How it works

One operator, one shared brain-state, two surfaces (CLI + dashboard):

Claude Code (operator)  ──drives──▶  yq CLI toolbelt
        │                                │
        │  reads instructions            │ writes state + logs activity
        ▼                                ▼
   ┌──────────────────  shared state  ──────────────────┐
   │  SQLite (positions, trades, equity, signals, news,  │
   │  journal, instruction inbox) + DuckDB candles       │
   └──────────────────────────┬──────────────────────────┘
                              ▼
              FastAPI cockpit + SPA dashboard
       (charts · positions · trade approvals · inbox · news)

The operator loop

  1. Read the inbox first — the user leaves instructions from the dashboard: yq inbox.
  2. Do the work with the toolbelt. Every command logs to shared state, so the dashboard updates live.
  3. Record decisions as trades (paper by default; live orders queue for human approval) and as journal entries for the next session.

Install & quickstart

pip install -e '.[all]'        # everything
pip install -e '.[web]'        # just the dashboard
pip install -e '.[binance]'    # Binance data/trading
pip install -e '.[feeds]'      # news/RSS + disclosures
# 1. collect candles
yq collect BTCUSDT 1d 1h

# 2. research
yq features BTCUSDT 1d
yq backtest BTCUSDT 1d macross --fast 5 --slow 20
yq optimize BTCUSDT 1d macross --walk-forward 4
yq brief BTCUSDT                              # one-screen research digest

# 3. operate the account
yq watch add BTCUSDT --interval 1d
yq risk set max_open_positions=5 daily_loss_limit=200
yq decide --weight 0.1 --execute             # signals → risk-sized paper orders
yq cycle                                     # refresh → scan → mark → notify

# 4. launch the cockpit
yq dashboard          # → http://127.0.0.1:8000

Information layer

Non-price inputs — the biggest edge for stocks. The key idea: collection is free (keyless public sources) and judgement is free too (the operator, Claude Code, reads the items and decides). No paid sentiment API.

News (RSS)

Keyless RSS/Atom reader tags watchlist symbols and stores headlines in the news table.

Sentiment

A keyword scorer (EN + KR finance terms) auto-tags each item; the operator can override.

Disclosures (DART)

전자공시 filings for Korean corporates (needs free DART_API_KEY).

Fundamentals (KIS)

Stock price + PER / PBR / EPS / BPS / market cap via 한국투자증권.

Strategies, backtesting & risk

Built-in strategies: macross, volatility_breakout, rsi_reversion, donchian_breakout — toggle on/off from the cockpit. Backtests support a full risk layer:

Backtest(candle, strategy, risk=RiskConfig(
    sizing="volatility", stop_loss=0.05,
    take_profit=0.1, max_drawdown=0.2))

At the account level, the risk policy (yq risk) is enforced on every order — paper or live — and can reject a trade before it fills. yq expect records a backtest baseline and yq decay warns when realized performance erodes below it.

Memory & persistence

The operator is ephemeral — a fresh Claude Code session has no recall of the last one, and on the web the container is reclaimed after inactivity. So anything worth keeping must live outside the session. YammyQuant uses the same memory tiers most agentic setups converge on:

TierWhereHolds
Semantic
stable knowledge		 CLAUDE.md (committed to git) Operating procedure, conventions, the toolbelt. Loaded into context every session.
Structured
facts & config		 yammyquant_state.db → settings table (yq … set) Risk limits, strategy toggles, target weights, flags like sentiment_gate.
Episodic
why you did things		 journal table (yq journal) Theses, entry/exit rationale, lessons — free-text, tagged, time-ordered.
World state
what is true now		 state DB tables + DuckDB candles Positions, trades, equity curve, signals, collected news, the inbox.
The rule of thumb
If it should change behavior in every future session → CLAUDE.md. If it's a tunable value → settings. If it's a decision and its reasoning → the journal. If it's an observation about the market or account → it's already a row in the state DB.

How most people manage agent memory

This mirrors common practice with agentic AI:

  • Project memory files — CLAUDE.md / AGENTS.md, committed to the repo, auto-loaded into context. The agent's "long-term instructions."
  • A durable store the agent reads/writes through tools — a database (here SQLite), a vector store for retrieval, or plain notes/markdown files. The point is the data outlives the chat context.
  • An append-only journal / decision log the agent re-reads at the start of a session — the pattern behind "memory stream" agents.
  • Summaries over raw history — don't try to keep every token; persist distilled state and reload that.

The anti-pattern is relying on the chat transcript as memory: it's capped, it's lost when the container is reclaimed, and it isn't shared with the dashboard. In YammyQuant, the transcript is disposable — the SQLite state DB, the DuckDB candles, and CLAUDE.md are the memory.

The memory stream — yq recall

Re-reading the whole journal every session doesn't scale, so the journal is a retrievable memory stream (the Generative-Agents pattern). Each note can carry an --importance (1–10); yq recall ranks notes by recency × importance × relevance and bundles unread inbox + open positions into one session-start digest:

yq journal "scaled into BTC on breakout, high conviction" --tag thesis --importance 8
yq recall                 # top memories + inbox + positions
yq recall "ETH merge"     # a query biases retrieval toward matching notes
SignalMeaning
recencyhalf-life decay (~14d) — recent notes weigh more.
importanceoperator-set salience (1–10); defaults to mid when unset.
relevanceterm overlap with the query (0 when no query is given).
First & last thing each session
Start with yq recall (memory + inbox + positions) and read CLAUDE.md. End by writing a yq journal … --importance N entry so next-you starts informed.

Money safety

Live trading has two hard gates
A live order executes only with (1) the env flag YQ_ALLOW_LIVE=1 and (2) explicit human approval (a dashboard button or yq approve). Without the flag, an approved live order is rejected. The operator never sets that flag — it's the user's call.

Paper trading is the default and fills immediately against a simulated book. Binance keys are read from BINANCE_API_KEY / BINANCE_SECRET_KEY (environment only — never hardcoded). Notifications (Discord webhook) fire when a live order needs approval, a risk rejection happens, or a cycle finds signals.

The library

The CLI is a thin layer over a clean, tested Python framework you can use directly:

from yammyquant import Candle, Backtest, MACross
result = Backtest(candle, MACross(5, 20), cash=10_000, fee=0.001).run()
print(result)   # Sharpe, max drawdown, win rate, profit factor, ...
